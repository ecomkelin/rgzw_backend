# rgzw_backend 数据库设计文档

本文档详细描述了 rgzw_backend 项目的数据库结构、实体关系和权限控制机制，旨在帮助开发者理解和维护系统架构。

## 1. 概述

rgzw_backend 是一个培训机构管理系统，主要功能包括账户管理、组织管理、课程管理、学生管理、课包销售和课堂教学等。

## 3. 详细实体关系

### 3.1 授权/账户模块 (Authorization)
- **Account (账号表)**: 存储系统中所有用户的基础账号信息
  - 一对多关系 → **Student**: 一个账号可关联多个学生身份
  - 一对多关系 → **User**: 一个账号可关联多个员工身份
  - 一对多关系 → **OrderPack**: 一个账号可购买多个订单

### 3.2 组织架构模块 (Organization)
- **Org (机构表)**: 存储培训机构的基本信息
  - 一对多关系 → **User**: 一个机构可有多个员工
  - 一对多关系 → **Student**: 一个机构可有多个学生
  - 一对多关系 → **Course**: 一个机构可开设多个课程
  - 一对多关系 → **Room**: 一个机构可有多个教室
  - 一对多关系 → **Subject**: 一个机构可开设多门科目

- **User (员工表)**: 存储机构员工信息（与账号关联）
  - 一对多关系 → **Course**: 一个老师可教授多个课程（作为主讲或助教）

- **Room (教室表)**: 存储机构的物理教室信息

- **Dept (部门表)**: 存储机构内部部门信息

### 3.3 课程体系模块 (School/Course)
- **Subject (科目表)**: 存储可开设的课程科目
  - 一对多关系 → **Course**: 一个科目可开设多个班级

- **Course (课程表)**: 存储具体的课程班级信息
  - 一对多关系 → **Lesson**: 一门课程有多个课次
  - 一对多关系 → **StudentCourse**: 一门课程可有多个学生报名

### 3.4 学生管理模块 (School/Student)
- **Student (学生表)**: 存储学生信息（与账号关联）
  - 一对多关系 → **StudentCourse**: 一个学生可报名多个课程
  - 一对多关系 → **StudentPack**: 一个学生可拥有多个课包

- **StudentCourse (学生课程关联表)**: 关联学生与课程的报名关系
  - 一对多关系 → **LessonAttendance**: 一个报名记录对应多个课次考勤

- **StudentPack (学生课包表)**: 存储学生拥有的课包信息

### 3.5 课包订单模块 (Pack)
- **Pack (课包表)**: 存储课包产品信息
  - 一对多关系 → **OrderPack**: 一个课包可有多个订单

- **OrderPack (课包订单表)**: 存储课包购买订单
  - 一对多关系 → **StudentPack**: 一个订单可生成多个学生课包

### 3.6 课堂管理模块 (School/Lesson)
- **Lesson (课次表)**: 存储每次上课的具体安排
  - 一对多关系 → **LessonAttendance**: 一节课有多个学生考勤
  - 一对多关系 → **LessonWork**: 一节课产生多个学生作品
  - 一对多关系 → **LessonEvaluation**: 一节课产生多个学生评价

- **LessonAttendance (课堂考勤表)**: 记录学生的课堂出勤
  - 一对一关系 → **StudentPack**: 考勤记录可能消耗课包

- **LessonWork (课堂作品表)**: 存储学生的课堂作品

- **LessonEvaluation (课堂评价表)**: 存储老师对学生的课堂评价

## 4. 权限控制逻辑

### 4.1 账号类型 (accountType)
- **User**: 员工账号
- **Student**: 学生账号

### 4.2 角色类型 (roleTemp)
- **manager**: 管理员（机构管理者）
- **teacher**: 教师（授课老师）

### 4.3 权限规则

#### 4.3.1 学生权限
- 只能查看自己的个人信息（Student表）
- 只能查看自己的报名记录（StudentCourse表）
- 只能查看自己的课包（StudentPack表）
- 只能查看自己的考勤记录（LessonAttendance表）
- 只能查看自己的课堂作品（LessonWork表）
- 只能查看自己的课堂评价（LessonEvaluation表）
- 可以查看开放的科目和招生中的课程（Subject表和Course表）

#### 4.3.2 教师权限
- 可以查看、修改自己授课课程的相关信息（Course表）
- 可以对学生进行考勤、评价、查看作品（LessonAttendance、LessonEvaluation、LessonWork表）
- 可以查看自己课程的报名情况（StudentCourse表）
- 可以查看自己课程的课次安排（Lesson表）

#### 4.3.3 管理员权限
- 可以管理所在机构的所有数据
- 可以增删改查所在机构的所有模块数据
- 可以管理所有员工（User表）
- 可以管理所有学生信息（Student表）
- 可以管理所有课程和课包（Course、Subject、Pack表）

#### 4.3.4 机构隔离规则
- 不同机构的数据完全隔离
- 用户只能操作自己机构内的数据
- 通过Org字段实现数据隔离

#### 4.3.5 管理员特权
- `isAdmin` 字段标识系统超级管理员
- 超级管理员可以跨机构操作数据
- 有些敏感操作仅限超级管理员执行

## 5. 关键业务逻辑

### 5.1 课包消耗逻辑
- 当LessonAttendance表中的lessonConsumed字段变为true时，系统会自动从关联的StudentPack中扣除课时
- 每次考勤消耗1课时（remainingLesson - 1）

### 5.2 数据一致性保障
- 使用Mongoose中间件保证Org表中isMain字段的唯一性
- 通过索引和唯一约束保证数据完整性
- 在DAO层进行权限验证，防止越权操作

### 5.3 审计跟踪
- 所有数据表都包含createdBy和updatedBy字段，用于追踪数据变更历史
- 使用timestamps插件自动管理createdAt和updatedAt字段

## 6. 扩展建议

1. **权限精细化**: 当前使用roleTemp字段简化权限管理，后续可考虑引入完整的RBAC权限系统
2. **数据归档**: 长期运行后，考虑对历史数据进行归档处理
3. **报表优化**: 针对报表查询需求，可考虑添加聚合索引
